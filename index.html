For the steps below, you can run Git commands in either the built-in terminal inside Visual Studio Code or the terminal provided by the Git tool you installed (Git Bash). Editing your code (creating or updating files) and deploying it (uploading it to the cloud) are separate stages — you don’t need to do both at the same time. A common beginner workflow looks like this:
1. Edit or create your project files in Visual Studio Code.
2. When you’re ready, open either Git Bash terminal or run Git commands on Visual Studio Code with git add, git commit, and git push to upload your work to GitHub.
This means you can close one terminal and open the other at any time — they work independently. You can switch between them whenever you like, depending on what feels more comfortable for you.

Basic Git Operations
After you finish creating the repository, editing your code, and working on your project, it’s time to update the latest version on GitHub. To do that, you’ll need to commit your changes and push the code to the cloud. You can also create another branch (e.g., dev) to temporarily store your work in progress before merging it into the main branch later.

Collaboration

Prerequisite: Before creating a pull request, you need to have created and worked on another branch (e.g., dev) and pushed your changes to GitHub. A PR is used when you want to merge that branch into the main (production) branch, and you need approval first.

Git collaboration happens through pull requests (PRs), which let you propose merging your code into another branch while teammates review it, leave comments, suggest fixes, and approve changes before merging. Local commands like git status help you track progress up to committing, but they don’t confirm whether code is pushed to the cloud. Commits, branches, PRs, and code reviews work together to support clear and safe teamwork in Git.

Merge Strategy
Conflicts: Someone else has pushed changes to the same branch or the main branch after your last pull. Because the code you pulled doesn't align with the changes pushed by someone, there is a mismatch. Then, when you try to merge or pull, Git will see different changes in the same part of a file and ask you to resolve conflicts.
You cannot commit when there are unresolved conflicts. First, resolve the conflicts in your files, stage the changes, and then commit to ensure the main branch stays correct and functional.
Example: If Alice changes line 10 and Bob also changes line 10 before pulling Alice’s update, Git will mark a conflict. You must then choose which change to keep.
